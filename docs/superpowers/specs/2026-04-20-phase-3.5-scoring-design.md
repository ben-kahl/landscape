# Phase 3.5 — Reinforcement-Aware Scoring

**Status:** Approved design, spec only — no implementation plan yet.
**Author:** brainstorming session 2026-04-20.
**Branch:** `phase-3.5-scoring`.
**Tracks tasks:** #23 reinforcement-aware ranking.

## Goal

Reshape the retrieval scoring function so that reinforcement — a well-worn entity or edge — actually amplifies ranking instead of adding a bounded bonus that rarely decides ties. Today, reinforcement is additive and weakly dominant; cold-start candidates with modest base similarity can outrank warm candidates with equal base similarity only by small margins. We want "well-trodden paths are easier to traverse" to be the operative behavior.

Two concrete changes:

1. **Multiplicative gating.** `score = base × (1 + γ·reinforcement)` instead of additive. Cold paths stay at base; warm paths get a bounded boost.
2. **Write-side reinforcement.** `upsert_entity` and `upsert_relation` bump `access_count` and `last_accessed` on every assertion — not just on query-side touches. An agent repeatedly asserting a fact reinforces it now, without needing an intervening retrieval.

## Non-goals

- **Confidence decay (task #24) is explicitly deferred.** Supersession already handles real temporal conflicts; adding a wall-clock decay on `edge_confidence` would damage timeless facts ("Alice was born in Portland") without a motivating benchmark case. Revisit only if a concrete failure demands it, likely as type-aware decay (temporal rel_types only) rather than blanket decay.
- Changing seed-search, graph-expansion, or extraction. Scoring only.
- Introducing a new reinforcement source beyond writes and queries. Dwell-time, explicit feedback, etc. are out of scope.

## Current state

`src/landscape/retrieval/scoring.py`:

```
score = α·vec_sim + β·proximity + γ·reinforcement + δ·edge_confidence
```

Defaults: α=1.0, β=0.8, γ=0.4, δ=0.3, reinforcement_cap=2.0, λ_reinf = ln(2)/7days.

- `reinforcement_score(access_count, last_accessed, now, weights)` already applies `log1p · exp(-λ·age)` capped at 2.0.
- Reinforcement is *additive*: a high-reinforcement cold-similarity candidate can outrank a fresh high-similarity candidate by a γ·cap margin, which is backwards for a relevance system.
- `access_count` is only bumped by `touch_entities` / `touch_relations` at query time. Writes don't reinforce.

## Design

### Multiplicative gating

```
base  = α·vec_sim + β·proximity + δ·edge_confidence
score = base × (1 + γ·reinforcement)
```

- `reinforcement` stays bounded by `weights.reinforcement_cap` (default 2.0).
- `γ` drops from 0.4 → 0.2 because it now multiplies, not adds.

**Bound invariant.** Max score = `(α+β+δ) · (1 + γ·cap)` = `2.1 · 1.4` = `2.94`. Finite, enforceable in tests.

**Why multiplicative.** The additive form says "a warm cold-similarity fact can beat a fresh hot-similarity fact." The multiplicative form says "reinforcement *amplifies* base relevance." Cold paths stay at base; warm paths get a bounded 1.0× to 1.4× boost. Matches the biological framing and is robust against rumination — a cold candidate at base=0 stays at 0 no matter how reinforced it is.

### Write-side reinforcement

Any assertion bumps reinforcement counters, symmetric with query-side `touch_entities` / `touch_relations`.

- `neo4j_store.upsert_entity(...)` → `SET e.access_count = coalesce(e.access_count, 0) + 1, e.last_accessed = $now`.
- `neo4j_store.upsert_relation(...)` → same on the `RELATES_TO` edge.
- Applies to all write paths by transitivity: `pipeline.ingest_document`, MCP `add_entity` / `add_relation` / `remember`.
- Same `+1` delta as query-side; reinforcement cap still applies.

**Motivation.** An agent repeatedly re-asserting "Alice works at Zylos" today should reinforce that fact *now*, without needing an intervening retrieval.

## Components and contracts

### `scoring.py`

Modified function:

```python
def score_candidate(
    vector_sim: float,
    graph_distance: int,
    edge_confidence: float,
    reinforcement: float,
    weights: ScoringWeights,
) -> float:
    proximity = 1.0 / (1.0 + max(0, graph_distance))
    base = (
        weights.alpha * clamp01(vector_sim)
        + weights.beta * proximity
        + weights.delta * clamp01(edge_confidence)
    )
    return base * (1.0 + weights.gamma * reinforcement)
```

Signature unchanged; only the combining formula changes.

Modified `max_possible_score`:

```python
def max_possible_score(weights: ScoringWeights) -> float:
    base = weights.alpha + weights.beta + weights.delta
    return base * (1.0 + weights.gamma * weights.reinforcement_cap)
```

`reinforcement_score` is unchanged.

### `neo4j_store.py`

`upsert_entity` adds to its Cypher:

```cypher
SET e.access_count = coalesce(e.access_count, 0) + 1,
    e.last_accessed = $now
```

`upsert_relation` adds the same on the `RELATES_TO` edge. Both are idempotent: `coalesce(...,0)+1` is safe on first write (where no counter exists) and on re-writes.

No cap enforcement in Cypher — the `log1p · cap` in scoring still bounds the contribution.

### `config.py`

```python
scoring_gamma: float = 0.2   # was 0.4 — now multiplicative
```

No new config. `decay_lambda` and `reinforcement_cap` unchanged.

## Data flow

### Query path (unchanged structure, new combining step inside `score_candidate`)

```
retrieve(query)
  → seed search
  → hydrate seeds
  → bfs_expand
  → for each candidate:
       score = score_candidate(vec_sim, distance, edge_conf, reinforcement, w)
         # internally: base × (1 + γ·reinforcement)
  → rank, reinforce, return
```

### Write path (new behavior)

```
upsert_entity(name, type, ...)
  → MERGE entity
  → SET access_count = coalesce(access_count,0) + 1, last_accessed = $now

upsert_relation(subject, rel_type, object, confidence)
  → MERGE edge (unchanged)
  → SET access_count = coalesce(access_count,0) + 1, last_accessed = $now
```

## Error handling

- **Rumination.** Reinforcement capped at 2.0; multiplier capped at 1.4. Max score bounded at ~2.94. Invariant test updated.
- **Cold start.** `reinforcement = 0` → multiplier = 1.0 → score == base. Additive→multiplicative migration doesn't penalize never-touched candidates.
- **Idempotency on write.** `coalesce(access_count, 0) + 1` is safe on first write (no existing counter) and on re-writes.

## Testing

### New unit tests in `tests/test_scoring.py`

- `test_multiplicative_reinforcement_amplifies`: two candidates with identical `vec_sim`, `proximity`, `edge_confidence`, only one with `reinforcement > 0`; reinforced one scores strictly higher.
- `test_cold_path_baseline`: `reinforcement = 0` → score equals base (multiplier exactly 1.0).
- `test_zero_base_stays_zero`: `vec_sim = 0`, `proximity = 0` (impossible in practice — but), `edge_conf = 0` → any reinforcement leaves score at 0. Confirms gating can't resurrect a truly irrelevant candidate.
- `test_max_score_invariant_under_gating`: pathologically hot input yields score ≤ `(α+β+δ)·(1+γ·cap) + 1e-9` and hits the ceiling exactly.
- `test_rumination_bound_preserved`: 10M access_count, very recent — reinforcement still ≤ cap; score still ≤ max_possible_score.

### New integration tests in `tests/test_reinforcement.py`

- `test_write_bumps_entity_access_count`: call `upsert_entity` three times on the same entity; `access_count == 3` in Neo4j.
- `test_write_bumps_relation_access_count`: call `upsert_relation` three times on the same `(subject, rel_type, object)`; edge `access_count == 3`.
- `test_reasserted_fact_outranks_cold_fact`: ingest two equally-similar facts into the graph; call `remember` (or `upsert_relation` directly) on one three extra times; query touching both; reinforced one ranks above cold one.

### Existing tests to update

- `tests/test_scoring.py::test_score_bounded_by_max` — update ceiling expectation to `(α+β+δ)·(1+γ·cap)`.
- `tests/test_scoring.py::test_score_components_balanced` — replace with a test appropriate to the multiplicative model. Specifically: `γ·cap ≤ 1.0` so the multiplier stays in `[1, 1.4]` and a reinforced candidate can amplify its base relevance by at most 40%.
- `tests/test_scoring.py::test_log1p_sublinear_growth`, `test_decay_half_life`, `test_reinforcement_is_bounded` — unchanged. They test `reinforcement_score`, which is unchanged.

### Benchmark regression

- `scripts/bench_retrieval.py` must still report 7/7 on the killer-demo corpus.
- Weight change (γ: 0.4 → 0.2) may shift individual MRR values. Acceptable if P@k holds; any P@k regression blocks merge and triggers re-tuning.
- PR body includes before/after MRR numbers.

## Migration and rollout

- No data migration. Existing entities/edges have `access_count = 0` / `last_accessed = null` → no reinforcement multiplier until first touch. Self-heals on first query or next write.
- No MCP contract change. Retriever output shape unchanged.
- No new config values. Only existing `scoring_gamma` default changes.

## Open questions (resolved during brainstorming)

- Write-side reinforcement uses the same `+1` bump as query-side. Rationale: simplest symmetric model; tune later if benchmark demands asymmetry.
- Entity and relation write-side reinforcement are symmetric. Rationale: asymmetry would feel inconsistent and we have no reason to favor one.
- Bound `(α+β+δ)·(1+γ·cap) = 2.94` is acceptable. Rationale: rumination invariant preserved; concrete number is irrelevant.
- Confidence decay (#24) is **deferred**. Supersession already handles temporal conflicts; blanket decay would damage timeless facts. Revisit with a motivating benchmark case, likely as type-aware decay.
