# Phase 3.5 — Reinforcement-Aware Scoring

**Status:** Approved design, spec only — no implementation plan yet.
**Author:** brainstorming session 2026-04-20.
**Branch:** `phase-3.5-scoring`.
**Tracks tasks:** #23 reinforcement-aware ranking, #24 recency decay on confidence.

## Goal

Reshape the retrieval scoring function so that two biologically motivated signals actually affect ranking:

1. **Reinforcement** — well-worn entities and edges (high `access_count`, recently touched) should outrank cold-start candidates of equal base relevance.
2. **Confidence recency** — an edge asserted 30 days ago at confidence 0.9 should rank below an edge asserted today at confidence 0.7, all else equal.

Today, reinforcement is additive and weakly dominant; confidence is time-independent. Both are insufficient to back the "biologically inspired reinforcement + temporal decay" framing in `CLAUDE.md`.

## Non-goals

- Changing the seed-search or graph-expansion logic. Scoring only.
- Changing embedding or extraction. Numeric-fact recall gaps in LongMemEval are an extraction issue and are out of scope.
- Introducing a new reinforcement source (e.g. dwell time, explicit feedback). Write-side reinforcement is the only new source.

## Current state

`src/landscape/retrieval/scoring.py`:

```
score = α·vec_sim + β·proximity + γ·reinforcement + δ·edge_confidence
```

Defaults: α=1.0, β=0.8, γ=0.4, δ=0.3, cap=2.0, λ = ln(2)/7days.

- `reinforcement_score(access_count, last_accessed, now, weights)` already applies `log1p · exp(-λ·age)` capped at 2.0.
- `edge_confidence` is passed through unchanged — no time dependency.
- Reinforcement is *additive*: a high-reinforcement cold-similarity candidate can outrank a fresh high-similarity candidate, which is the wrong direction for a relevance system.

## Design

### Two-timescale decay with multiplicative gating

```
decayed_conf = edge_conf · exp(-λ_conf · age_seconds(edge.valid_from, now))
base         = α·vec_sim + β·proximity + δ·decayed_conf
score        = base × (1 + γ·reinforcement)
```

- `λ_conf = ln(2) / (30·86400)` — 30-day half-life on edge confidence.
- `λ_reinf` stays at `ln(2) / (7·86400)` — 7-day half-life on reinforcement activation.
- `reinforcement` stays bounded by `weights.reinforcement_cap` (default 2.0).
- `γ` drops from 0.4 → 0.2 because it now multiplies, not adds.

**Bound invariant.** Max score = `(α+β+δ) · (1 + γ·cap)` = `2.1 · 1.4` = `2.94`. Finite, enforceable in tests.

**Why multiplicative.** The additive form says "a warm cold-similarity fact can beat a fresh hot-similarity fact." The multiplicative form says "reinforcement *amplifies* base relevance." Cold paths stay near base; warm paths get a bounded boost. This matches the biological framing and is robust against rumination.

**Why two timescales.** Semantic confidence ≠ episodic activation. A fact stated once and never revisited should remain trustworthy for weeks; a recently-touched entity should remain primed for days. One decay constant cannot serve both.

### Write-side reinforcement

Any assertion bumps reinforcement counters, symmetric with query-side `touch_entities` / `touch_relations`.

- `neo4j_store.upsert_entity(...)` → `SET e.access_count = coalesce(e.access_count, 0) + 1, e.last_accessed = $now`.
- `neo4j_store.upsert_relation(...)` → same on the `RELATES_TO` edge.
- Applies to all write paths by transitivity: `pipeline.ingest_document`, MCP `add_entity` / `add_relation` / `remember`.
- Same `+1` delta as query-side; reinforcement cap still applies.

**Motivation.** An agent repeatedly re-asserting "Alice works at Zylos" today should reinforce that fact *now*, without needing an intervening retrieval.

## Components and contracts

### `scoring.py`

New function:

```python
def confidence_score(
    edge_confidence: float,
    valid_from: datetime | None,
    now: datetime,
    weights: ScoringWeights,
) -> float:
    """Decayed edge confidence. Bounded in [0, 1].

    If valid_from is None (legacy rows), returns edge_confidence with no decay."""
```

- Clamps `edge_confidence` to `[0, 1]` first.
- Returns `edge_confidence` if `valid_from` is None.
- Returns `edge_confidence · exp(-λ_conf · max(0, age_seconds))` otherwise.

Modified function:

```python
def score_candidate(
    vector_sim: float,
    graph_distance: int,
    decayed_confidence: float,   # was: edge_confidence
    reinforcement: float,
    weights: ScoringWeights,
) -> float:
    proximity = 1.0 / (1.0 + max(0, graph_distance))
    base = (
        weights.alpha * clamp01(vector_sim)
        + weights.beta * proximity
        + weights.delta * clamp01(decayed_confidence)
    )
    return base * (1.0 + weights.gamma * reinforcement)
```

Parameter rename from `edge_confidence` → `decayed_confidence` is a **breaking change to the scoring API**. `query.py` is the only internal caller; tests will be updated.

Modified `max_possible_score`:

```python
def max_possible_score(weights: ScoringWeights) -> float:
    base = weights.alpha + weights.beta + weights.delta
    return base * (1.0 + weights.gamma * weights.reinforcement_cap)
```

### `query.py`

For each candidate:

1. Compute path-level `avg_conf` as today.
2. Compute path-level `valid_from` as the *min* of `valid_from` across edges in the path — the path is only as current as its oldest edge.
3. Call `confidence_score(avg_conf, min_valid_from, now, w)`.
4. Pass the result as `decayed_confidence` to `score_candidate`.

Seed candidates (distance 0) have no edge; `decayed_confidence = 0.0` as today.

`bfs_expand` must project `valid_from` alongside the existing `edge_confidences`, `edge_access_counts`, `edge_last_accessed`. Verified: today it projects confidences and access counts but **not** `valid_from`. Add `[r IN rels | r.valid_from] AS edge_valid_from` to the projection. No schema change; `valid_from` is already persisted on `RELATES_TO`.

### `neo4j_store.py`

`upsert_entity` adds to its Cypher:

```cypher
SET e.access_count = coalesce(e.access_count, 0) + 1,
    e.last_accessed = $now
```

`upsert_relation` adds the same on the edge. Both are idempotent and append-only for the counter. No cap is enforced in Cypher — the `log1p · cap` in scoring handles bounding.

### `config.py`

```python
confidence_decay_lambda: float = math.log(2) / (30 * 86400)
scoring_gamma: float = 0.2   # was 0.4
```

Legacy `decay_lambda` is renamed `reinforcement_decay_lambda` for clarity, with a backwards-compatible alias if any other module imports it. (Grep confirms only `scoring.py` reads it; no alias needed.)

## Data flow

### Query path

```
retrieve(query)
  → seed search (unchanged)
  → hydrate seeds (unchanged)
  → bfs_expand (now projects valid_from)
  → for each expansion row:
       decayed_conf = confidence_score(avg_conf, min_valid_from, now, w)
       score = score_candidate(vec_sim, distance, decayed_conf, reinforcement, w)
  → rank, reinforce, return
```

### Write path

```
upsert_entity(name, type, ...)
  → MERGE entity
  → SET access_count = coalesce(access_count,0) + 1, last_accessed = $now

upsert_relation(subject, rel_type, object, confidence)
  → MERGE edge
  → SET valid_from = $now (unchanged semantics for new edges)
  → SET access_count = coalesce(access_count,0) + 1, last_accessed = $now
```

## Error handling

- **Missing `valid_from` on legacy edges.** `confidence_score` returns raw `edge_confidence`, no decay. Graceful degradation; logs nothing.
- **Clock skew / negative age.** `max(0, age_seconds)` in `confidence_score` so negative ages collapse to no decay.
- **Rumination.** Reinforcement still capped at 2.0; multiplier capped at 1.4. Max score bounded at ~2.94. Invariant test updated.
- **Idempotency on write.** `coalesce(access_count, 0) + 1` is safe on first write (no existing counter) and on re-writes.

## Testing

### New unit tests in `tests/test_scoring.py`

- `test_confidence_score_fresh_is_undecayed`: `valid_from == now` → returns `edge_confidence`.
- `test_confidence_score_30_day_half_life`: `valid_from = now - 30d` → returns `edge_confidence / 2 ± 1e-9`.
- `test_confidence_score_none_valid_from_no_decay`: returns `edge_confidence` untouched.
- `test_confidence_score_clamps_input`: out-of-range `edge_confidence` clamps to `[0,1]` before decay.
- `test_multiplicative_reinforcement_amplifies`: two candidates with identical `vec_sim`, `proximity`, `decayed_conf`, only one with `reinforcement > 0`, the reinforced one scores strictly higher.
- `test_cold_path_baseline`: reinforcement = 0 → score == base. Multiplier is exactly 1.
- `test_max_score_invariant_under_gating`: pathologically hot input yields score ≤ `(α+β+δ)·(1+γ·cap) + 1e-9` and hits the ceiling exactly.
- `test_rumination_bound_preserved`: 10M access_count, very recent — reinforcement still ≤ cap.

## Migration and rollout

- No data migration. Existing entities and edges have `access_count = 0` / `last_accessed = null` → no reinforcement bonus until first touch, self-heals.
- Existing `RELATES_TO` rows already write `valid_from` (confirmed in `upsert_relation`); legacy rows without it fall to graceful-degradation.
- Weight change (γ: 0.4 → 0.2) lives in `config.py` and is already parameterized per-settings; no env churn required.
- No MCP contract change. Retriever output shape unchanged.

## Open questions (resolved during brainstorming)

- `valid_from` is the age source for confidence decay (not `extracted_at`). Rationale: re-ingesting the same fact should reset its freshness.
- Write-side reinforcement uses the same `+1` bump as query-side. Rationale: simplest possible symmetric model; tune later if benchmark demands.
- Entity and relation write-side reinforcement are symmetric. Rationale: the feature would feel inconsistent if one but not the other.
- Bound `(α+β+δ)·(1+γ·cap) = 2.94` is acceptable. Rationale: the rumination invariant is preserved; concrete number is irrelevant.
